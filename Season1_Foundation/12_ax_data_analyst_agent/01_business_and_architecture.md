# 📊 Code Interpreter 에이전트의 비즈니스 가치와 아키텍처

## 1. 정적 도구 사용(Static Tool Calling)의 한계
이전 11번째 모듈에서는 에이전트가 개발자가 미리 짜둔 `check_inventory()` 같은 함수만 골라서 썼습니다. 
이 방식의 치명적인 단점은, 사장님이 *"재고 데이터랑 생산량 데이터를 엮어서 새로운 꺾은선 그래프를 그려봐"* 라고 예외적인 지시를 내리면, 개발자가 새로운 파이썬 함수(`draw_custom_graph`)를 짜주기 전까지는 에이전트가 아무것도 할 수 없다는 점입니다.

## 2. 동적 코드 실행 (Dynamic Code Interpreter)의 마법
데이터 사이언티스트 에이전트는 함수를 미리 짜두지 않습니다. 대신 모델에게 **"너가 파이썬 코드를 텍스트로 짜주면, 내가 내 컴퓨터에서 실행해줄게"**라는 권한을 줍니다.
*   에이전트는 사용자의 변덕스러운 지시 사항에 맞춰 그때그때 `pandas` 코드를 창조해 냅니다.
*   **가장 위대한 특징 (Self-Correction):** 인간 프로그래머도 코드를 짜면 버그가 납니다. 에이전트가 짠 코드를 컴퓨터가 실행했을 때 뻘건색 에러(Traceback)가 뜨면, 엔진은 당황하지 않고 그 에러 텍스트를 복사해서 LLM에게 다시 말을 겁니다. 
    > *"너가 짠 코드 돌려봤더니 `KeyError: 'date'` 에러가 떴어. 컬럼명이 잘못된 거 같은데 고쳐봐."*
    그러면 LLM이 코드를 수정해서 다시 보냅니다. 이 디버깅 루프가 에이전트를 진정한 지능체로 만듭니다.

## 3. 보안 이슈 (Sandboxing)
LLM에게 내 컴퓨터의 커맨드 라인 실행 권한을 주는 것은 극도로 위험합니다. LLM이 악의적인 프롬프트 인젝션에 당해 `os.system("rm -rf /")` 같은 코드를 짜버리면 서버가 날아갑니다.
➔ **실무 해결책:** 기업 환경에서는 LLM이 짠 코드를 메인 서버가 아닌, 인터넷이 차단된 일회용 **Docker 컨테이너(Sandbox)** 안에서만 실행하고 결과를 받아오는 격리(Isolation) 아키텍처를 반드시 설계해야 합니다.
